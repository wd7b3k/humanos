# Релизы текстов протоколов HumanOS

Набор **снимков (релизов)** — это версионирование пользовательских текстов практик: тело, дыхание, фокус, поддержка для каждого состояния. Код бота и движок протоколов не меняются — меняется только JSON, который читается при старте.

## Зачем это нужно

- править формулировки без деплоя кода (достаточно файла + реестра);
- хранить историю и при необходимости **откатиться** на прошлый снимок;
- явно описывать **суть изменений** в массиве `notes` (это же читается в админке).

Это **не** релизы приложения в магазинах: это данные контента на сервере.

## Где лежат файлы

| Путь | Назначение |
|------|------------|
| `data/protocol_releases/registry.json` | Активный `active_release_id` и список `release_ids` |
| `data/protocol_releases/releases/<id>.json` | Полный снимок текстов одного релиза |
| `data/runtime/release_events.jsonl` | Журнал событий (bootstrap, смена активного релиза) |

Формат JSON снимка задаётся загрузчиком `build_protocol_engine_from_release()` в `domain/protocol_engine.py`: поля `release_id`, `version`, `title`, `notes`, `created_at`, `protocols` с вариантами и шагами (`how_to`, `alternatives`, `goal`, `notes`, `duration_seconds`, …).

## Как добавить новый релиз

1. Скопировать существующий файл из `releases/` как основу.
2. Поменять `release_id`, `version`, `title`, `notes`, `created_at` и нужные тексты в `protocols`.
3. Положить файл: `data/protocol_releases/releases/<release_id>.json`.
4. Добавить `<release_id>` в массив `release_ids` в `registry.json` (порядок — как в реестре; в админке список показывается с новыми сверху).
5. Чтобы сделать релиз **активным**: либо вручную выставить `active_release_id` в реестре и перезапустить бота, либо использовать **админку** (см. ниже).

После смены активного релиза процесс бота должен перечитать данные — в штатном сценарии админка запускает перезапуск `humanos-bot.service`.

## Админка в Telegram

Доступна пользователям из `ADMIN_IDS`: раздел **«Аналитика»** → кнопка **«Релизы текстов протоколов»**.

Там показывается:

- какой релиз **сейчас активен**;
- список всех снимков с датой из JSON и заголовком `title`;
- **журнал** последних событий (действия подписаны по-русски; в файле `release_events.jsonl` остаются машинные коды `bootstrap`, `activate`).

Кнопки со списком релизов: **✅** — активный снимок, **📦** — архивный. Открытие карточки показывает:

- идентификатор и версию;
- статус на понятном русском;
- **состав протоколов** (состояние, внутренний ключ, название протокола из JSON);
- **суть изменений** — нумерованный список из поля `notes`.

Неактивный релиз можно сделать активным кнопкой **«Сделать этот релиз активным»** — в журнал попадёт запись с пометкой о переключении из админки, затем выполняется перезапуск сервиса.

## Связанные документы

- `docs/PROTOCOL_EVIDENCE.md` — внутренняя аргументация по формулировкам и техникам.
- Экспорт текстов в виде сообщений бота: `docs/PROTOCOL_FULL_BOT_TEMPLATE_RU.md` (генерируется из активного релиза скриптом/однократным запуском).

## Важно для эксплуатации

- Релизы **иммутабельны по смыслу**: лучше добавить новый файл, чем править старый, если нужна история отката.
- Несколько инстансов бота с общим `data/` должны видеть один и тот же реестр; иначе «активный» релиз разъедется между машинами.
